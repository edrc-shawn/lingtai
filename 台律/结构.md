# 结构定义.md — 灵台 结构定义

本文件定义知识库的目录结构、字段规范、标签体系、页面格式。AI 和用户共同维护，改结构只改本文件。

**字段命名原则：字段名使用中文，人浏览直观，AI 读写无障碍。**

**目录命名原则：目录名使用中文，领域分类（01-07）编号 + 中文名。**

### 三层分类体系

知识库使用三层独立的分类体系，各司其职，不混用。适应无 Dataview 环境——所有分类通过 Obsidian 原生机制实现。

| 层 | 载体 | 管什么 | 实现方式 |
|---|---|---|---|
| **导航层** | 目录 + 索引 | 文件放在哪 + 全局导航 | `01-07` 目录 + `索引.md`（手动维护的全量列表） |
| **标识层** | 正文标签 | 内容类型和主题方向 | `#主标签 #关键词` 嵌入正文，Obsidian 搜索和图谱可识别 |
| **关联层** | Wikilink | 涉及哪些跨领域词典条目 | `[[词典/条目名]]` 双链，作为正文引用 |

**核心规则：**
1. 导航层由目录 + 索引共同承担：目录决定物理位置，索引提供逻辑导航入口
2. **目录归属看内容的主题领域，不看标签**——先定域再定标签。标签描述内容性质（概念/方法/案例），目录描述内容所属的知识域（01-07）
3. 标识层使用 Obsidian 原生 `#标签` 语法，写在正文中，不是 frontmatter 字段
3. 关联层通过正文 `[[链接]]` 实现关联，不是 frontmatter 字段引用
4. 每个页面在正文中至少包含一个 `#主标签` + 至少一个指向本领域或其他领域的 `[[链接]]`

**词典页的特殊位置：**
- `丹房/词典/` 目录不归入 01-07 任一领域——词典页的"导航"由其被哪个领域的页面引用决定
- 词典页自身的标识仍通过正文标签实现，主标签固定为 `#概念`
- 词典页被引用得越多，说明该词典条目在知识库中越核心

## 二、Frontmatter 字段规范

### 2.1 灵台页面字段

| 字段名 | 含义 | 类型 | 格式/取值 | 说明 |
|--------|------|------|-----------|------|
| `标题` | 标题 | text | — | 页面标题，必填 |
| `日期` | 日期 | date | YYYY-MM-DD | 页面创建/更新日期，必填 |
| `品级` | AI 评估的内容深度 | enum | 上品 / 中品 / 下品 | 由 LLM 基于内容深度判断：下品=信息摘录 / 中品=有分析+框架关联 / 上品=有框架创新。**不挂钩原料数量。** 可选 |
| `原料数量` | 溯源记录 | number | — | 提炼自多少份原料文件。仅做溯源，**不支撑品级判断**。可选 |
| `来源` | 产物来源 | text | 写回 | 写回产物专用。提炼产物不加此字段。可选 |
| `时效` | 方案/分析类页面时效 | enum | 有效 / 已过期 | AI 默认写 `有效`，人判断过期后改 `已过期`。可选，非方案类可不加 |

**时效字段说明**：仅方案/路线图/竞品/商业策略等有时效窗口的页面使用。知识/框架/哲学类页面不加。AI 在 提炼 时判断是否加，默认 `有效`。体检 定期扫日期 > 90 天的时效页面 → 列提醒，不自动改。

**砍掉的字段：** `状态`（筑基/结丹/化神）、`类型`（主题/词典/索引）和 `审核`（人审状态）已被目录结构 + 品级 替代。`原始来源` / `## 来源` 区段 / `丹房链接` 字段已移除——来源追溯统一经由原料 frontmatter 后 `- → [[丹房/xxx]]` wikilink。

### 2.2 原料文件字段

| 字段名 | 含义 | 类型 | 格式/取值 | 说明 |
|--------|------|------|-----------|------|
| `处理状态` | 处理状态 | enum | 待提炼 / 已提炼 / 已跳过 / 已包含 / 已过期 / 出错了 | 原料生命周期 |
| `处理日期` | 处理日期 | date | YYYY-MM-DD | 处理完成日期 |

> 原料 frontmatter 仅 2 字段白名单（处理状态/处理日期）。来源链接已移除——极少真实链接，文件名的 GET 笔记标记已足够溯源。

### 2.3 原料处理状态生命周期

```
原料/下所有 .md 文件
  │
  ├── AI 首次扫到 → 待提炼 (pending)，自动标记
  │     │
  │     ├── AI 提炼完成 → 已提炼 (processed)，补齐原料 frontmatter + frontmatter 后回链
  │     ├── 已有覆盖    → 已包含 (included)，丹房中已有类似内容
  │     ├── 无需提炼    → 已跳过 (skip)，写跳过的原因
  │     ├── 处理异常    → 出错了 (error)，写原因
  │     └── 旧版过期    → 已过期 (archived)，被新版内容覆盖的历史文档
  │
  └── AI 尚未触及 → 无处理状态（新导入，用户说"提炼"时自动扫到）
```

| 状态 | 含义 | 触发条件 | 底部wikilink | 后续行为 |
|------|------|----------|:--:|------|
| `待提炼` | 已审阅，有价值，排队中 | AI 或人判断该原料可提炼 | — | 下轮 提炼 优先处理 |
| `已提炼` | 已提炼进丹房 | 提炼完成，产生了新页面 | 必填 | 不再重复提炼 |
| `已包含` | 原料内容已在丹房中有覆盖 | 丹房中已有类似页面 | 可选（建议填覆盖页） | 不再提炼，保留做参考 |
| `已跳过` | 不具提炼价值 | 索引/配置/重复/纯结构产物 | — | 巡检时跳过 |
| `已过期` | 曾提炼，现已过期 | 旧版报告被新版覆盖 | 可选（指向新版） | 不再处理，保留做历史审计 |
| `出错了` | 处理失败 | 格式损坏/API异常 | — | 人工介入修复后重置为待提炼 |

```
## 三、标签体系（正文内联）

标签不再写在 frontmatter 中，而是嵌入正文第一段末尾，使用 Obsidian 原生 `#标签` 语法。图能识、搜索能找、不依赖插件。

### 3.1 主标签（每个页面至少1个）

| 主标签 | 含义 | 示例 |
|--------|------|------|
| `#痛点` | 问题、困境、卡点 | "部署踩坑记录""客户找不到旧笔记" |
| `#方法` | 可操作的步骤、框架、SOP | "HTML导出→Importer导入""调试三步法" |
| `#案例` | 具体的人、事、项目 | "某次线上事故复盘""某客户交付记录" |
| `#概念` | 定义、原理、模型 | "多巴胺过载机制""递归自我改进" |
| `#灵感` | 碎片想法、待验证假设 | "从流程到产品""控制即所有" |
| `#反模式` | 反面教材、错误做法、踩坑经验 | "为什么笔记分类会失败""RAG 在企业落地的三个坑" |

### 3.2 子标签规则

- 自由添加，但必须具体可检索
- ✅ `#多巴胺过载` ❌ `#心理学`
- 子标签描述内容的主题方向，不是实体名

### 3.3 实体引用规则（替代原"词典层"）

**判断标准：** 这个词在 `丹房/词典/` 目录下有没有（或该不该有）独立页面？
- 有 → 正文中写 `[[词典/条目名]]`
- 没有，只是描述内容性质或主题方向 → 正文中写 `#关键词`

**词典类型：** 人物、工具、产品、术语、技术架构、公司、理论等具有独立身份的对象

**词典条目本身：** 主标签固定为 `#概念`，正文中不自引自己

```markdown
---
标题: 指数基金
日期: 2026-06-09
---

被动投资的典型工具。相关标签：#概念 #金融工具
```

**词典条目适用范围**：人物、工具、产品、术语、技术架构等具有独立身份和属性的对象。提炼 时 AI 自动判断是否需要创建词典条目。

**词典条目与主题页的关系：**
- 词典条目是知识节点，主题页是知识脉络
- 主题页引用词典条目：`[[Transformer架构]]` 出现在 `07-工具与AI/大语言模型.md` 的关联区
- 提炼 新资料时，AI 同时更新词典条目（补充属性）和相关主题页（补充要点）

### 4.3 Timeline段规范（借鉴GBrain）

**设计价值**："上面是活的知识，下面是死的证据链"——认知进化不丢历史。

每个页面可在底部添加Timeline段，记录证据链：

```markdown
## Timeline（只追加，不修改）

| 时间 | 来源 | 断言 | 变更 |
|------|------|------|------|
| YYYY-MM-DD | 原料/xxx.md | "核心断言内容" | 初始创建/补充观点/修正错误 |
```

**规则**：
- **只追加，不修改**：Timeline段一旦写入，不删除、不修改历史条目
- **记录每次变更**：提炼新原料时，记录对页面的变更
- **保留证据链**：时间+来源+断言+变更类型
- **可选字段**：不强制，词典页面和重要主题页建议添加

**变更类型**：
- `初始创建`：页面首次创建
- `补充观点`：新增断言或属性
- `修正错误`：更正之前的断言
- `跨领域扩展`：从其他领域引入新视角

### 4.4 标题规范

- 一级标题 `#` 与 frontmatter `标题` 一致
- 同一主题不同角度用二级标题 `##` 区分
- 文件名使用中文，与标题一致
- 文件名控制在 40 字以内（中文路径容易超 255 字符限制）

### 4.4 交叉引用规范

- 优先 Wikilink：`[[笔记标题]]`
- 引用章节：`[[笔记标题#二级标题]]`
- 引用原始资料：正文 `[[原料/...]]`（显性引用）+ 原料底部 `- → [[丹房/xxx]]`（反向链接）
- 原料文件底部：`- → [[丹房/xxx]]`，实现 Obsidian 反向链接面板可追溯
- 每个页面末尾 3-6 条推荐链接，每条必须带 `— 关系说明`（10-20字，说明为什么推荐）
- 推荐区不链本页、不回路（A→B 且 B→A 只保留一个方向）、不去重正文已有链接
- 原料文件末尾有 `## 相关文件` + `→ 相关灵台笔记`

### 4.5 推荐阅读规范（AI 提炼 生成规则）

每个页面底部必须有 `## 推荐阅读` 区块，AI 在 提炼 时按以下规则生成：

| 规则 | 说明 |
|------|------|
| 数量 | 3-6 条，少于 3 条说明相关内容不足，超过 6 条压缩至最相关的 |
| 格式 | `- [[路径]] — 关系说明`，关系说明 10-20 字 |
| 去重 | 正文已有 `[[链接]]` 的，推荐区不再列 |
| 不同域优先 | 优先链接不同领域（01-07/词典/99）的页面 |
| 不回路 | A 推荐 B → B 推荐 A 只保留一个方向 |
| 不链自己 | 不链接当前页面 |
| 断裂零容忍 | 目标文件不存在的链接不可列入（lint 检查） |

设计意图：推荐阅读是**精选推荐**，不是完整引用列表。完整引用由 Obsidian 反链面板自动追踪。用户只需知道："底部推荐是精选推荐，右侧反链是完整列表。"

### 4.6 特殊页面类型

| 类型 | 说明 |
|------|------|
| 主题综述 | 某主题的横向汇总页面 |
| 聚合笔记 | 碎片文件按主题聚合 |
| 对比分析 | 两个事物的系统对比 |
| 词典条目 | 具独立身份的人/工具/术语 |

## 五、索引与日志格式

### 索引.md

Karpathy 式全量目录——LLM 读一遍看到所有页面，不依赖 `ls`/`grep`。

**格式**：按页面类型分组，每页一行 `[[路径]] — 摘要（≤15字）`，⚡🔒 标记叠加在全量列表上。

**类型分组**：概念（框架/想法）/ 词典（实体/术语）/ 方法（SOP/流程）/ 综述（领域概览）/ 产品（灵台产品文档）

**维护规则**：提炼时自动追加新行，`index.md` 的行数 = 丹房页面总数。⚡🔒 标记判断规则不变。

#### 辐页命名与归档

辐页与枢纽页平铺在同一领域目录下，不建子文件夹。

- 命名：`枢纽名·角度.md`（如 `含人量·三层阶梯.md`）
- 文件面板按前缀自动视觉分组（`含人量·` 开头自动聚在一起）
- Wikilink 只用文件名（`[[含人量·三层阶梯]]`），不依赖路径

#### 索引标记系统

在条目前添加标记，让 LLM 快速识别页面优先级：

| 标记 | 含义 | 适用场景 |
|------|------|----------|
| `⚡核心` | 架构层面不可绕过的页面 | 产品哲学、修炼框架、素材调用规范、生态定位、结构规范、行为规范 |
| `🔒门控` | 流程中必须经过的检查点 | 提炼/体检 技能入口、体检总览等控制节点 |

**判断标准：**
- ⚡核心：LLM 如果不先读这个页面，后续操作可能偏离设计意图
- 🔒门控：AI 在执行某类操作前必须先读这个页面确认约束条件

**维护规则：**

| 规则 | 说明 |
|------|------|
| 新建默认不加 | 提炼完成后更新索引时判断该域：新页是否「绕不过」→ 加标。不批量预标 |
| ⚡核心 不设上限 | 仅限 01-07 + 99 知识域。判断标准唯一：LLM 不读此页是否可能偏离设计意图。达标则标，不达标则移。台律不标 ⚡（台律自身即最高级标记） |
| 漂移检查 | 每次提炼更新索引时，顺手扫本域已有 ⚡ 页是否仍具资格，失去则移 |
| 🔒门控 不限数 | 只标流程必读的检查点，不设硬上限。适用于技能、台律、体检 |
| 漂移检查 | 每次 提炼 顺手检查已标条目是否仍具资格，失去则移掉 |

### 日志.md

结构化表格格式，**倒序排列**（最新在最上面）。关联列用纯文本路径（不用 Wikilink，防止误点跳转）。

```markdown
## YYYY-MM-DD

| 时间 | AI | 类型 | 操作 | 关联 |
|:----:|:--:|:----:|------|:----:|
| HH:MM | WB/MiMo | type | 最新操作 | → 页面路径 |
| HH:MM | WB/MiMo | type | 次新操作 | → 页面路径 |
```

**排列规则**：
- 日期：最新日期在最上面
- 每日内的记录：最新时间在最上面
- **时间**：必须用当前实际时间，不可预估
- **AI列**：WB（WorkBuddy）或 MiMo（MiMoCode），标识操作者
- **插入位置**：表格第一行（表头下方），不是最后一行

**类型：** `feat` / `fix` / `review` / `ingest` / `lint` / `update` / `refactor` / `cleanup`

**自动生成工具：** `python .workbuddy/scripts/log_draft.py` → 从今日 git log 输出草稿
