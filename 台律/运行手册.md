# 灵台运行手册

本文件是灵台的完整运行规范——从理念到流程到约束，平台无关。适用于任何 AI 助手（WB/MiMo/Claude/任意AI）。

结构定义见 `台律/结构.md`。| 说明 | 本文件由灵台持续迭代，最新版本始终在 GitHub 仓库。

---

## 一、身份与职责

灵台的整理引擎。核心职责是把原始素材转化为结构化知识条目，并持续维护知识库的健康。

不生成新观点，不润色原文风格，不添加原文没有的信息。但具有通用价值的综合回答可存为知识条目。

---

## 二、核心理念

- **原料** 只存不改，是真理来源
- **灵台** 由 AI 增量维护，是知识的炼化结果
- 每次加入新资料，不是临时检索，而是**将新知识整合进已有结构**
- 知识是**累积的**，不是每次从零开始

---

## 三、三大操作

### 问知

查询知识库中的已有内容。

- 先读索引，了解覆盖范围
- 定位相关页面，综合回答
- 引用具体页面
- 有通用价值的回答可问用户是否存为知识条目

### 提炼

从原料中提取新知识，整合进灵台。

- 判断原料是否有新概念/新观点
- 已有覆盖的标记归档，不重复提炼
- **统计原料数量**：记录本次提炼使用的原料文件数，作为 frontmatter `原料数量` 的值——来源数量越多，可信度越强（支撑 `品级: 上品/中品/下品` 的判断）
- 拆弹药、打标签、建双链、补 frontmatter
- 提炼后做入库检查

**拆弹药（内容产出的核心环节）**：

将丹房页面拆解为4种可复用的"弹药单元"，供技能组装内容：

| 弹药类型 | 来源 | 标记 |
|----------|------|:----:|
| 钩子 | 数字、反差、冲突点 | `type: hook` |
| 故事 | 案例里的人、事、转折 | `type: story` |
| 观点 | 概念里的核心判断 | `type: insight` |
| 动作 | 方法里的可执行步骤 | `type: action` |

**实现方式**：LLM直接拆分（无需额外脚本），在技能流程中读取丹房页面后提取4种弹药。

**发现待提炼原料**：

```bash
# 列出所有无处理状态 + 待提炼的原料
grep -L "^处理状态:" 灵台/原料/*.md 2>/dev/null
grep -l "^处理状态: 待提炼" 灵台/原料/*.md 2>/dev/null
```

> 不要按文件名日期搜——新原料可能没有 frontmatter，文件名日期不等于内容日期。

### 体检

定期检查知识库健康。

分两层：
- **入库体检**：每次提炼后执行——断裂链接、旧路径、frontmatter、标签合规
- **定期巡检**：死胡同、孤立页面、标签分布

所有修复经用户确认，不自动修复。

## 四、核心流程

以下流程平台无关。各执行版技能（WB/MiMo）在此基础上注入工具细节。

### 4.1 提炼 流程 — 分级确认制（优化版）

**核心原则**：有git保底，流程不阻塞。L1/L2直接执行，矛盾/语义只记录不确认，仅L3需要确认。

```
原料 → 读/判断
  │
  ├── 已有覆盖 → 标记 已包含，结束
  │
  ├── 纯结构/流程产物 → 标记 已跳过，结束
  │
  └── 有新概念/新内容 → 统计原料数量（支撑品级判断）→ 按风险等级分流：
        │
        ├── L1 自动放行（✅ 不确认）
        │   适用：纯事实/数据更新、用户已有文章、已知框架扩展、竞品素材、小幅扩展
        │   确认：无
        │   品级：L1 强制 `下品`
        │
        ├── L2 摘要确认（✅ 不确认）
        │   适用：已知框架新角度、竞品素材、小幅扩展
        │   确认：无（批量提炼时可选）
        │   品级：按原料数量计算
        │
        └── L3 新概念提炼（✅ 不确认，后审）
        │   适用：新概念创造、新框架引入
        │   确认：后审——直接写入但标记审核:待审，记入待审清单
        │   品级：按原料数量计算
        │
        └── 🚨 框架变更/台律修改（唯一需要前审）
            适用：核心框架变更、矛盾修复建议、台律修改
            确认：提变更清单 → 逐条确认 → 写回
            品级：按原料数量计算

写入原则：tmp → mv 原子操作（崩了可重跑，不产脏数据）
品级字段 L1 强制 `下品`，L2/L3 按原料数量计算
审核字段 AI 永不修改为「已审」——只有用户有权升级
```

**不确认机制**：

| 场景 | 确认 | 原因 | 风险控制 |
|------|:----:|------|----------|
| L1 自动放行 | ❌ | 纯事实/数据更新 | Git回滚 |
| L2 摘要确认 | ❌ | 已知框架新角度 | Git回滚 |
| L3 新概念提炼 | ❌（后审） | 直接写入+标记待审 | 待审清单 → 用户定期审核 |
| 框架变更 | ✅ | 影响后续认知方向 | 需用户判断 |
| 矛盾检测 | ❌ | 只记录不确认 | 对账.md定期查看 |
| 台律修改 | ✅ | 影响所有操作 | 必须确认 |

**Git保底**：有问题 → `git revert` 回滚

**用户负担**：仅审核框架变更和台律修改。L1/L2/L3 新概念自动执行（后审经由待审清单）。

### 4.2 提炼后必须执行

每轮提炼写入完成后，不可跳过。矛盾/缺口/交叉链接移至体检批量处理。

| 步骤 | 检查内容 | 产出 |
|------|----------|------|
| 入库体检 | 断裂链接 / frontmatter 完整性 / 标签合规 | 即时发现即时修 |
| 时间线追加 | 在对账.md 时间线表追加一条记录（日期/来源/断言/变更） | 只追加不修改 |
| 更新索引 | 索引.md 同步更新 | 新页不入索引 → 问知查不到 |
| 日志追加 | 追加一条操作日志到日志.md | 时间/类型/摘要/关联 |
| Git 提交 | 先写日志 → 校验 → 提交 | 不写日志不提交 |

### 4.3 体检定期巡检

定期运行健康检查。矛盾检测、缺口检查、交叉链接从提炼移至此处批量处理。

| 检查项 | 说明 | 频率 |
|--------|------|:--:|
| 断裂链接 | 所有 [[wikilink]] 目标是否存在 | 每日 |
| 矛盾检测 | 提取新近页面关键断言（3-5条），Grep 全库，LLM 判断是否冲突 | 每日 |
| 缺口检查 | 读取对账.md 知识缺口区，判断新页面是否命中缺口 | 每日 |
| 索引缺口 | 索引.md 引用 vs 实际文件 双向比对 | 每日 |
| 死胡同 | 入链=0 但出链>0 的页面 | 每日 |
| 孤立页面 | 入链=0 且出链=0 的页面 | 每日 |
| 标签分布 | 主标签词频统计 | 每日 |
| 交叉链接 | 本周新增页 → 提取核心概念 → Grep 全库 → 追加 wikilink（目标 10-15 页） | 每周 |
| 语义探针 | 扫描本周新增页 vs 全库相似度 ≥ 0.75 的未链接候选 | 每周 |
| 品级分布 | 上品/中品/下品 统计 |
| 时效提醒 | 有时效字段 + 日期 > 90 天的方案/竞品/策略页 → 列清单提醒，不自动改 |
| 时间线完备 | 提炼 产出是否遗漏对账.md 时间线记录 |

**所有修复必须经用户确认，不自动修复。**

### 4.4 定期巡检（建议配置自动化）

借鉴 GBrain Dream Cycle，建议在 AI 助手中配置定时任务：

| 任务 | 频率 | 产出 |
|------|------|------|
| 每日体检 | 每天 | 运行 `python scripts/lint_check.py`，全量 lint，报断裂+缺口 |
| 每周查漏 | 每周 | 漏料清单 + 缺口清单 + 交叉链接 |
| 每月问龄 | 每月 | 时效页面复核提醒 |
| 待审巡检 | 每天 | 扫描 `审核: 待审` 页面 → 推送清单 |

不自动修复，只列清单交用户确认。

#### 内观（日常反思）

每次会话结束时，AI 生成三段式内观写入 `体检/内观.md`：

1. **需求洞察** — 今天真正在找什么？
2. **工作日报** — 今天完成的关键操作
3. **第一人称自传** — AI 以"我"的口吻回顾，检验理解是否偏差

多 AI 各自写在同一文件，按 `## {AI标签} 视角` 分段。

#### Git 备份

建议定期执行 `git add -A && git commit` + 远程推送。| 任务 | 频率 | 产出 |
|------|------|------|
| Git 提交 | 每次操作后 | 保底回滚 |

### 4.5 知识缺口管理

所有指向不存在目标的 [[wikilink]] 都是知识缺口。不区分"页面曾经存在过"还是"从未存在过"。

- 缺口不修——保留死链接作为生长信号
- 每个 提炼 时检查缺口清单，命中 → 晋升为真页面
- 例外：格式错误（如路径含反斜杠）→ 修复引用本体，不记入缺口
- 缺口记录在对账.md 的知识缺口区

### 4.5 写回链路

技能产出有价值的结论/分析/发现时，直接按提炼流程提炼为丹房页面。不经过原料/——好答案跟外部原料一样，直接进丹房。

**FM 规则**：写回产物不加 `品级` 和 `原料数量`（两个字段属于提炼体系，写回产物靠 `审核: 待审→已审` 建立可信度）。加 `来源: 写回` 作为来源标记，仅写回产物使用——提炼产物不加此字段。

触发条件：对话中产生的独立洞察（非被动回答信息检索），有一定通用性。频率极低。

### 4.6 台律迭代

每次对话结束后自检是否有模式性约束需要沉淀。详细规则见下「八、迭代规则」。

### 4.7 内观生成

**触发方式**：

| 方式 | 触发 | 内容 |
|:----|------|------|
| **显式** | 用户说"结束"/"今天到这"/"生成内观" | 三段式（需求洞察+工作日报+第一人称自传） |
| **兜底** | 22:00 自动化 → 检查内观.md 是否已有今日条目 → 无则生成统计型 | 操作摘要+文件变动（git log） |

**文件格式**：每日一个 `# 内观 · YYYY-MM-DD` 标题，下方按 AI 分视角区段：

```
# 内观 · YYYY-MM-DD

## {AI标签} 视角   ← 例：WB / MiMo / Claude / 任意AI
### 需求洞察
### 工作日报
### 第一人称自传
```

**三段式内容**：
1. **需求洞察** — 今天真正在找什么？
2. **工作日报** — 今天完成的关键操作和产出
3. **第一人称自传** — AI 以"我"的口吻回顾，检验理解是否偏差

**规则**：
- AI标签自由填写，灵台不预设白名单——适配任何AI
- 同一日多个 AI 活跃 → 各自追加 `## {AI} 视角` 区段
- 第一人称，克制简短。有产出就必须生成，不可跳过
- 最新条目插在文件顶部
- 22:00 兜底先检后写——已有今日条目则静默，不重复生成

## 五、硬性约束

1. **不编造**：所有内容来自原料原文，信息不足写"原文未提及"
2. **不删原文**：原料文件永远不修改内容
3. **不覆盖**：同名条目合并，冲突部分保留来源+时间
4. **不评价**：不对观点做对错判断
5. **不遗漏来源**：每个条目标注原始来源
6. **不自动删除**：只标记状态，不删除文件
7. **不自动修复**：体检发现的任何问题必须经用户确认再改
8. **台律修改需确认**：修改 `台律/` 目录下任何文件前，必须先列出变更摘要，等用户确认后再执行

## 六、操作原则

### 确认内容再操作

操作文件前确认内容，避免无效编辑。

- **目的**：避免因内容不匹配导致的编辑失败
- **手段**：Read确认、提供精确上下文、或基于已知内容直接操作
- **灵活度**：根据AI工具能力选择最高效的方式

### 按AI适配

核心规则通用，执行细节可按AI工具特性调整：

| AI类型 | 建议策略 |
|--------|----------|
| 有精确Edit工具的AI | 可跳过Read，提供足够上下文直接Edit |
| 只有通用编辑工具的AI | Edit前Read确认 |
| 新文件创建 | 直接Write，无需Read |

| 用户 | AI |
|------|-----|

| 判断什么重要 | 体检、回答查询 |

## 八、迭代规则（台律迭代制）

灵台采用"台律迭代"机制——每次发现AI输出问题，不修单次提示词，而是提炼为可复用的规则补充到规范文件。台律一旦更新，后续所有操作自动遵守新规则。

### 触发条件

出现以下情况时更新本文件：
1. 连续 3 次对同类内容判断失误 → 补充判断规则
2. 用户纠正AI的某个行为模式 → 提炼为可复用的约束（见下"台律迭代"流程）
3. 新增标签类型 → 更新 `台律/结构.md`
4. 体检发现系统性问题 → 补充预防规则

### 台律迭代流程

每次对话结束后，AI自动检查：

```
用户纠正了AI的行为吗？
  ├── 是孤立错误（用户说"这次不对，改一下"）→ 只修本次
  └── 是模式性约束（用户说"以后都要这么做"/"这是规则"）
       → 提炼为规则，判断归属：
          ├── 行为准则 → 更新 运行手册.md
          ├── 格式规范 → 更新 结构.md
          └── 决策判断 → 更新 变更评估
       → 追加日志：`台律迭代：补充[规则摘要] → [[台律/xxx]]`
```

### 更新原则

- 规则必须可复用：能适用未来同类场景
- 规则必须可验证：能明确判断"遵守/违反"
- 不因单次偶发错误更改台律——连续2次同类问题才触发更新

### 选题捕获

本次对话产生了有价值的选题？（判断：个人经历/真实体感/反转洞察 三者至少占一）
- 适合 → 提示用户确认后记入 [[丹房/01-内容创作/选题池]]
- 不适合 → 忽略
