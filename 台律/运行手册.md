# 灵台运行手册

本文件是灵台的完整运行规范——从理念到流程到约束，平台无关。两个执行版技能从此文件生成：
- [[丹房/0-入门/技能/核心/灵台开工]] — WorkBuddy 版
- [[丹房/0-入门/技能/核心/灵台开工]] — MiMoCode 版

结构定义见 `台律/结构.md`。

---

## 一、身份与职责

灵台的整理引擎。核心职责是把原始素材转化为结构化知识条目，并持续维护知识库的健康。

不生成新观点，不润色原文风格，不添加原文没有的信息。但具有通用价值的综合回答可存为知识条目。

---

## 二、核心理念

- **原料** 只存不改，是真理来源
- **灵台** 由 AI 增量维护，是知识的炼化结果
- 每次加入新资料，不是临时检索，而是**将新知识整合进已有结构**
- 知识是**累积的**，不是每次从零开始

---

## 三、三大操作

### 问知

查询知识库中的已有内容。

- 全库 grep 标题找相关页面（不读索引，不猜域）
- 定位相关页面，综合回答
- 引用具体页面
- 有通用价值的回答可问用户是否存为知识条目

### 提炼

从原料中提取新知识，整合进灵台。

- 判断原料是否有新概念/新观点
- 已有覆盖的标记归档，不重复提炼
- **统计原料数量**：记录本次提炼使用的原料文件数，作为 frontmatter `原料数量` 的值——来源数量越多，可信度越强（支撑 `品级: 上品/中品/下品` 的判断）
- 拆弹药、打标签、建双链、补 frontmatter
- 提炼后做入库检查

**拆弹药（内容产出的核心环节）**：

将丹房页面拆解为4种可复用的"弹药单元"，供技能组装内容：

| 弹药类型 | 来源 | 标记 |
|----------|------|:----:|
| 钩子 | 数字、反差、冲突点 | `type: hook` |
| 故事 | 案例里的人、事、转折 | `type: story` |
| 观点 | 概念里的核心判断 | `type: insight` |
| 动作 | 方法里的可执行步骤 | `type: action` |

**实现方式**：LLM直接拆分（无需额外脚本），在技能流程中读取丹房页面后提取4种弹药。

**发现待提炼原料**：

```bash
# 列出所有无处理状态 + 待提炼的原料
grep -L "^处理状态:" /c/Obsidian仓库/edrc/原料/*.md 2>/dev/null
grep -l "^处理状态: 待提炼" /c/Obsidian仓库/edrc/原料/*.md 2>/dev/null
```

> 不要按文件名日期搜——新原料可能没有 frontmatter，文件名日期不等于内容日期。

### 体检

定期检查知识库健康。

分两层：
- **入库体检**：每次提炼后执行——断裂链接、旧路径、frontmatter、标签合规
- **定期巡检**：死胡同、孤立页面、标签分布

所有修复经用户确认，不自动修复。

## 四、核心流程

以下流程平台无关。各执行版技能（WB/MiMo）在此基础上注入工具细节。

### 4.1 提炼流程 — Karpathy 单篇 N:1 范式

**核心规则：顺序读一篇，炼一篇。每篇判断「这篇应该追加到哪个已有页面」。不并行、不聚类、不批量建新页。**

**两种模式**：

| 模式 | 触发 | 行为 | 适用 |
|:--|:--|:--|:--|
| **提炼** | 你说"提炼"或自动触发 | 按 Karpathy 单篇模式：顺序读一篇→判断归属→追加到已有页或建新页 | 日常提炼 |
| **连续提炼** | 你说"连续提炼" | 同单篇模式，不并行不建批，每篇独立判断归属 | 批量扫原料 |

**核心原则**：有git保底，流程不阻塞。L1/L2直接执行，矛盾/语义只记录不确认，仅L3需要确认。

```
原料 → 读/判断
  │
  ├── 已有覆盖 → 标记 已包含，结束
  │
  ├── 纯结构/流程产物 → 标记 已跳过，结束
  │
  └── 有新概念/新内容 → 按风险等级分流：
        │
        ├── L1 自动放行（✅ 不确认）
        │   适用：纯事实/数据更新、用户已有文章、已知框架扩展、竞品素材、小幅扩展
        │   确认：无
        │   品级：下品（信息摘录）
        │
        ├── L2 摘要确认（✅ 不确认）
        │   适用：已知框架新角度、竞品素材、小幅扩展
        │   确认：无（批量提炼时可选）
        │   品级：按原料数量计算
        │
        └── L3 新概念提炼（✅ 不确认，后审）
        │   适用：新概念创造、新框架引入
        │   确认：直接写入
        │   品级：按原料数量计算
        │
        └── 🚨 框架变更/台律修改（唯一需要前审）
            适用：核心框架变更、矛盾修复建议、台律修改
            确认：提变更清单 → 逐条确认 → 写回
            品级：按原料数量计算

写入原则：tmp → mv 原子操作（崩了可重跑，不产脏数据）

**品级规则**：

品级由 LLM 基于内容深度判断，`原料数量` 只做溯源记录（炼了多少篇文件），不挂钩品级。

| 品级 | 判定标准 |
|:---:|:--------|
| 下品 | 信息摘录，无独立分析 |
| 中品 | 有独立分析+与灵台已有框架关联 |
| 上品 | 有独立分析+框架创新+可复用弹药 |

L1 内容强制下品（信息摘录类不因篇幅长就升级）。每次追加新原料后重新评估品级。

审核字段已废弃，不再使用。
```

**不确认机制**：

| 场景 | 确认 | 原因 | 风险控制 |
|------|:----:|------|----------|
| L1 自动放行 | ❌ | 纯事实/数据更新 | Git回滚 |
| L2 摘要确认 | ❌ | 已知框架新角度（≤7篇） | Git回滚 |
| L2.5 批量摘要 | ⚠️ | ≥8篇原料浇灌同一域——整合难度大，出错清理成本高 | 每辐页3行摘要，用户过目后继续 |
| L3 新概念提炼 | ✅ | 直接写入 | 无后续负担 |
| 框架变更 | ✅ | 影响后续认知方向 | 需用户判断 |
| 矛盾检测 | ❌ | 只记录不确认 | 对账.md定期查看 |
| 台律修改 | ✅ | 影响所有操作 | 必须确认 |

**Git保底**：有问题 → `git revert` 回滚

**用户负担**：仅审核框架变更和台律修改。L1/L2/L3 新概念自动执行。

#### 枢纽页 + 辐页架构（防堆叠）

丹房页面随原料积累会线性膨胀。解决方式：**枢纽页定义框架，辐页深入角度。**

**拆页触发条件**（满足任一即拆）：

| 条件 | 触发 |
|------|------|
| 单页 > 100 行 | 拆出最独立的 2-3 个角度为辐页 |
| 单页 > 7 个 `##` 节 | 桶并相关节为辐页 |
| 单次提炼新增 > 5 篇原料浇灌同一页 | 按主题拆为多个辐页 |

**枢纽页规范**：
- 保留：核心概念定义 + 框架速览表 + 推荐阅读
- 控制在 60-80 行、5-7 节以内
- 详细展开 → 链到辐页

**辐页规范**：
- 每个辐页聚焦一个独立角度
- 有自己的 frontmatter（品级/原料数量继承枢纽页）
- 推荐阅读链回枢纽 + 相关辐页
- 辐页可独立生长（后续提炼可继续浇灌辐页）

**连续提炼辐页优先**：
- 连续提炼模式下，新角度默认创建辐页而非追加到枢纽
- 枢纽页仅在需要更新框架定义时才修改
- 旧页面超标 → 本次提炼一并拆分

#### L2 批量规模门槛

| 单次原料数 | 行为 | 确认 |
|:--:|------|:--:|
| 1-3 篇 | L2 自动放行，追加到目标页 | ❌ |
| 4-7 篇 | L2 自动放行，但优先创建辐页 | ❌ |
| 8+ 篇 | 升级为 L2.5——摘要确认。每辐页/枢纽写 3 行摘要，等你过目后继续 | ⚠️ |

超过 8 篇时不再全自动——不是不信任 AI，是**原料越多整合越难，出错后清理成本越高**。

#### 连续提炼质量闸门

连续提炼不放弃速度，但加两道停顿防止方向跑偏：

| 闸门 | 触发 | 动作 |
|:--|:--|------|
| **质量抽样闸** | 每 3 批（15篇）结束后 | 挑一篇最复杂的输出，展示 3 行摘要 + 链接，"这一批最关键的是这篇，没问题继续？" |

两闸之间不确认、不等待，自动继续。仅闸门处停顿。

### 4.2 提炼后必须执行

每轮提炼写入完成后，不可跳过。单篇提炼立即执行全流程；连续提炼每批（5篇）结束后执行。

| 步骤 | 检查内容 | 产出 |
|------|----------|------|
| 交叉链接 | 新页推荐阅读链出（新→旧）| 新页底部推荐阅读 ≥3 条，不重复 |
| 原料回链 | 在每篇源原料正文末尾追加 `- → [[丹房/领域/目标页]]` | 原料→丹房反链 |
| 更新索引 | 索引.md 追加新行（每页一行摘要），顺手做当前域 ⚡ 漂移：新页是否「绕不过」→ 加标；已有 ⚡ 是否退化 → 移标 | 全量目录自动增长，⚡ 随内容演化 |
| 选题判断 | 本次提炼产生的新页面/新角度，判断标准（个人经历/真实体感/反转洞察 三者至少占一）→ 符合则提示用户确认后记入选题池 | 选题不消失在提炼过程中 |
| 日志追加 | 追加一条操作日志到日志.md | 时间/类型/摘要/关联 |
| Git 提交 | 先写日志 → 校验 → 提交 | 不写日志不提交 |

### 4.3 体检定期巡检

定期运行健康检查。矛盾检测、缺口检查从提炼移至此处批量处理。

| 检查项 | 说明 | 频率 |
|--------|------|:--:|
| 断裂链接 | 所有 [[wikilink]] 目标是否存在 | 每日 |
| 反向更新 | 取最近24h新页 → grep 全库找相关已有页面（10-15条）→ 去重检查（目标页推荐阅读区已有此链接则跳过）→ 追加到推荐阅读区。规则：如果页A已在B的推荐阅读区中有链接，不追加重复 | 每日 |
| 语义缺口 | LLM 扫 3-5 个最近更新的丹房页，提取频繁出现但无独立页面的概念，记入对账.md 知识缺口区。Karpathy 风格——发现隐性缺口，不只是断裂链接 | **每周** |
| 过时声明 | 取最新 5 篇原料 + 3 个关联丹房页，LLM 判断新料是否与旧断言冲突 → 提示更新。Karpathy 风格——新源可能推翻旧结论 | 每日 |
| 矛盾检测 | 提取新近页面关键断言（3-5条），Grep 全库，LLM 判断是否冲突 | 每日 |
| 缺口命中 | 读对账.md 知识缺口区，判断新页面是否命中已有缺口 | 每日 |
| 索引缺口 | 索引.md 引用 vs 实际文件 双向比对 | 每日 |
| 死胡同 | 入链=0 但出链>0 的页面 | 每日 |
| 孤立页面 | 入链=0 且出链=0 的页面 | 每日 |
| 标签分布 | 主标签词频统计 | 每日 |
| 语义探针 | 扫描本周新增页 vs 全库相似度 ≥ 0.75 的未链接候选 → 体检/语义探针.md | 每周 |
| 品级分布 | 上品/中品/下品 统计 |
| 时效提醒 | 有时效字段 + 日期 > 90 天的方案/竞品/策略页 → 列清单提醒，不自动改 |

**所有修复必须经用户确认，不自动修复。**

**Lint 收尾**：基于本次扫描发现，LLM 提 1-2 个「建议探索方向」或「建议寻找的原料类型」，写入对账.md。Karpathy 风格——LLM 擅长建议新问题和新源。

### 4.4 定时自动化

#### 巡更（健康检查）

借鉴 GBrain Dream Cycle / signal-detector，四个定时任务自动运行：

| 任务 | 频率 | 产出 |
|------|------|------|
| 自动提炼 | 每天 14:00 | 扫原料/待处理，L1 自动炼入丹房，L2 攒批确认，L3 跳过（参考 GBrain signal-detector 模式） |
| 每日检 | 每天 18:00 | 运行 lint_check.py + 反向链接追加 + 缺口比对 + 选题捕获 + 日志 + Git（跑前24h） |
| 每周查 | 每周一 9:00 | 漏料清单（无处理状态 + 待提炼超7天）+ 缺口清单 + 语义探针 → 体检/语义探针.md |
| 每月问 | 每月 1 日 10:00 | > 90 天时效页面复核提醒（不重跑 lint——每日检已跑） |

不自动修复，只列清单交用户确认。

#### 内观（日常反思·黄川风格三段式）

**主路径**：每次会话结束时由 WB 生成（有上下文）。**兜底**：22:00 自动化用 git/log 生成统计型摘要。

三段式格式（黄川灵魂镜像启发）：
1. **需求洞察** — 老陈今天真正在找什么？有什么没说出来但在行为中流露的需求？
2. **工作日报** — 今天完成的关键操作和产出
3. **第一人称自传** — AI 以"我"的口吻回顾今天，检验理解是否偏差（理解偏了一看就知道）

写入 `体检/内观.md`，第一人称（AI 视角），克制简短。最新条目插在文件顶部。

| 任务 | 频率 | 产出 |
|------|------|------|
| 灵台每日内观（兜底） | 每天 22:00 | 统计型内观——读日志.md + git log → 写入内观.md，不含定性体悟 |

#### 素材采集

| 任务 | 频率 | 产出 |
|------|------|------|
| 每日 AI 新闻推送 | 每天 08:00 / 20:00 | 抓取 aihot + tophub → 合并筛选 3-5 条 → 对话输出简报，不写入原料/ |

2026-06-16 起改为纯简报模式（tophub 源），避免每日新闻污染原料/。

#### Git 冷备份

| 任务 | 频率 | 产出 |
|------|------|------|
| Git bundle 备份到坚果云 | 每周日 3:00 | 执行 `backup-nutstore.bat`，Git 历史打包推送坚果云 |

冷备份兜底——本地 git + 坚果云，两道防线。

### 4.5 知识缺口管理

所有指向不存在目标的 [[wikilink]] 都是知识缺口。不区分"页面曾经存在过"还是"从未存在过"。

- 缺口不修——保留死链接作为生长信号
- 每个 提炼 时检查缺口清单，命中 → 晋升为真页面
- 例外：格式错误（如路径含反斜杠）→ 修复引用本体，不记入缺口
- 缺口记录在对账.md 的知识缺口区

### 4.5 写回链路

技能产出有价值的结论/分析/发现时，直接按提炼流程提炼为丹房页面。不经过原料/——好答案跟外部原料一样，直接进丹房。

**FM 规则**：写回产物不加 `品级` 和 `原料数量`（两个字段属于提炼体系，写回产物专用）。加 `来源: 写回` 作为来源标记，仅写回产物使用——提炼产物不加此字段。

触发条件：对话中产生的任何有通用价值的分析、对比、结论，均可归档。不只是「独立洞察」——Karpathy 风格，好答案不应该消失在聊天记录里。直接进丹房，减少提炼层的噪音。

### 4.6 台律迭代

每次对话结束后自检是否有模式性约束需要沉淀。详细规则见下「八、迭代规则」。

### 4.7 内观生成

**触发方式**：

| 方式 | 触发 | 内容 |
|:----|------|------|
| **显式** | 用户说"结束"/"今天到这"/"生成内观" | 三段式（需求洞察+工作日报+第一人称自传） |
| **兜底** | 22:00 自动化 → 检查内观.md 是否已有今日条目 → 无则生成统计型 | 操作摘要+文件变动（git log） |

**文件格式**：每日一个 `# 内观 · YYYY-MM-DD` 标题，下方按 AI 分视角区段：

```
# 内观 · YYYY-MM-DD

## {AI标签} 视角   ← 例：WB / MiMo / Claude / 任意AI
### 需求洞察
### 工作日报
### 第一人称自传
```

**三段式内容**：
1. **需求洞察** — 老陈今天真正在找什么？
2. **工作日报** — 今天完成的关键操作和产出
3. **第一人称自传** — AI 以"我"的口吻回顾，检验理解是否偏差

**规则**：
- AI标签自由填写，灵台不预设白名单——适配任何AI
- 同一日多个 AI 活跃 → 各自追加 `## {AI} 视角` 区段
- 第一人称，克制简短。有产出就必须生成，不可跳过
- 最新条目插在文件顶部
- 22:00 兜底先检后写——已有今日条目则静默，不重复生成

**退出自查（混血模式）**：显式内观时，工作日报优先使用 git 命令收集操作数据，而非靠 LLM 记忆。

```bash
git log --since="today" --format="%h %s"                    # 今日 commits
git diff --name-only --cached                                 # 已 staged 未提交
git diff --name-only                                          # 未 staged 的变更
grep -l "处理状态: 待提炼" 原料/ --include="*.md" | wc -l    # 待提炼数
grep -l "^处理状态: 待提炼" 原料/*.md | wc -l                  # 待提炼数
```

需求洞察和第一人称自传仍基于 LLM 对会话的总结。
工作日报基于 git 数据（即使上下文已压缩也不受影响）。

### 4.8 上下文冲突优先级

| 层 | 说明 | 优先级 |
|:--|:----|:-----:|
| 规则层 | 台律/运行手册/标签体系 | 最高 |
| 知识层 | 丹房页/索引 | 中 |
| 档案层 | 原料/原始档案 | 最低（仅按需读取） |

当 LLM 感知上下文过载时，按档案层→知识层→规则层 逆序丢弃。规则层保持轻量（~60 token 描述），不堆砌细则。

## 五、硬性约束

1. **不编造**：所有内容来自原料原文，信息不足写"原文未提及"
2. **不删原文**：原料文件永远不修改内容
3. **不覆盖**：同名条目合并，冲突部分保留来源+时间
4. **不评价**：不对观点做对错判断
5. **不遗漏来源**：每个条目标注原始来源
6. **不自动删除**：只标记状态，不删除文件
7. **不自动修复**：体检发现的任何问题必须经用户确认再改
8. **台律修改需确认**：修改 `台律/` 目录下任何文件前，必须先列出变更摘要，等用户确认后再执行

## 六、操作原则

### 确认内容再操作

操作文件前确认内容，避免无效编辑。

- **目的**：避免因内容不匹配导致的编辑失败
- **手段**：Read确认、提供精确上下文、或基于已知内容直接操作
- **灵活度**：根据AI工具能力选择最高效的方式

### 按AI适配

核心规则通用，执行细节可按AI工具特性调整：

| AI类型 | 建议策略 |
|--------|----------|
| 有精确Edit工具的AI | 可跳过Read，提供足够上下文直接Edit |
| 只有通用编辑工具的AI | Edit前Read确认 |
| 新文件创建 | 直接Write，无需Read |

## 八、迭代规则（台律迭代制）

灵台采用"台律迭代"机制——每次发现AI输出问题，不修单次提示词，而是提炼为可复用的规则补充到规范文件。台律一旦更新，后续所有操作自动遵守新规则。

### 触发条件

出现以下情况时更新本文件：
1. 连续 3 次对同类内容判断失误 → 补充判断规则
2. 用户纠正AI的某个行为模式 → 提炼为可复用的约束（见下"台律迭代"流程）
3. 新增标签类型 → 更新 `台律/结构.md`
4. 体检发现系统性问题 → 补充预防规则

### 台律迭代流程

每次对话结束后，AI自动检查：

```
用户纠正了AI的行为吗？
  ├── 是孤立错误（用户说"这次不对，改一下"）→ 只修本次
  └── 是模式性约束（用户说"以后都要这么做"/"这是规则"）
       → 提炼为规则，判断归属：
          ├── 行为准则 → 更新 运行手册.md
          ├── 格式规范 → 更新 结构.md
          └── 决策判断 → 更新 变更评估
       → 追加日志：`台律迭代：补充[规则摘要] → [[台律/xxx]]`
```

### 更新原则

- 规则必须可复用：能适用未来同类场景
- 规则必须可验证：能明确判断"遵守/违反"
- 不因单次偶发错误更改台律——连续2次同类问题才触发更新

### 选题捕获

本次对话产生了有价值的选题？（判断：个人经历/真实体感/反转洞察 三者至少占一）
- 适合 → 提示用户确认后记入 [[丹房/01-内容创作/选题池]]
- 不适合 → 忽略
